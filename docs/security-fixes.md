# 개발 과정에서 발견한 보안 약점과 수정 내역

이 프로젝트는 처음부터 시큐어 코딩을 염두에 두고 작성했지만(비밀번호 해시, 파라미터
바인딩, CSRF 토큰, 입력 검증 등을 기능 구현과 동시에 적용), 실제로 기능을 서로 연결하고
테스트하는 과정에서 아래와 같은 구체적인 약점들을 발견해 수정했다. "예시 코드 없이
처음부터 만들어도 보안 요소를 놓치기 쉽다"는 강의 취지에 맞춰, 실제로 겪은 문제 위주로
기록한다.

## 1. 세션 고정 방지 도입이 CSRF 보호를 깨뜨린 문제

- **증상**: 로그인/회원가입 성공시 세션 고정(session fixation) 공격을 막기 위해
  `req.session.regenerate()`로 세션 ID를 재발급하도록 구현했는데, 그 직후 첫 요청이
  전부 403(CSRF 오류)으로 실패했다.
- **원인**: CSRF 토큰(`csrf-csrf`)이 세션 ID에 해시로 묶여 있는데, 세션이 재발급되며
  기존 `csrf-token` 쿠키는 새 세션 ID 기준으로는 더 이상 유효하지 않게 되었다.
- **Before**: 세션 재발급 후 이전 CSRF 쿠키를 그대로 둠 → 다음 요청에서 검증 실패.
- **After**: CSRF 토큰 발급 미들웨어(`src/app.js`)에서 `generateToken()` 검증 실패를
  `try/catch`로 감지해, 실패시 새 세션 기준으로 토큰을 즉시 재발급하도록 수정
  (`src/middleware/csrf.js`). 로그인/로그아웃/회원가입 전 시나리오를 curl로 재검증했다.

## 2. helmet 기본 CSP가 인라인 이벤트 핸들러를 막아 삭제 확인이 조용히 깨짐

- **증상**: 상품/유저/신고 삭제 버튼에 `onsubmit="return confirm(...)"` 인라인 핸들러를
  사용했는데, helmet의 기본 CSP(`script-src 'self'`)가 인라인 스크립트(속성 포함)를
  차단해 확인 다이얼로그가 뜨지 않았다. CSRF나 인가 로직 자체는 안전했지만, 우리가 이미
  적용해 둔 보안 헤더와 UI 코드가 서로 충돌한 사례였다.
- **Before**: `onsubmit="return confirm('삭제하시겠습니까?');"` 인라인 속성.
- **After**: `data-confirm` 커스텀 속성 + 외부 스크립트(`public/js/confirm-forms.js`)의
  `addEventListener`로 교체. 동시에 인라인 `style="..."` 속성도 CSS 클래스로 옮기고,
  helmet CSP의 `style-src`에서 `unsafe-inline`을 명시적으로 제거해 CSP를 더 엄격하게
  만들었다.

## 3. multer 1.x의 알려진 취약점

- **증상**: 초기 의존성 설치시 `npm install`이 `multer@1.4.5-lts.2`에 대해
  "다수의 취약점이 있으며 2.x에서 패치됨"이라는 경고를 출력했다.
- **Before**: `multer: ^1.4.5-lts.1`
- **After**: `multer: ^2.0.1`로 즉시 상향하고 업로드 기능(확장자/크기 제한 포함)을
  재검증했다. `npm audit` 결과 0 vulnerabilities로 확인.

## 4. 관리자의 유저 삭제가 연관 데이터로 인해 서버 에러를 일으킬 뻔한 문제

- **증상**: 상품/거래내역/메시지/신고가 있는 유저를 관리자가 삭제하면, FK 제약
  (`PRAGMA foreign_keys = ON`)에 의해 SQLite가 예외를 던진다. 이를 그대로 두면
  관리자 페이지에서 500 에러가 나면서 내부 에러 메시지가 노출될 위험이 있었다.
- **Before**: 삭제 요청을 그대로 실행 → 예외 발생시 전역 에러 핸들러가 500 렌더링.
- **After**: `src/routes/admin.js`에서 삭제를 `try/catch`로 감싸, 실패시 "연관된
  데이터가 있어 삭제할 수 없습니다. 휴면 처리를 이용하세요"라는 안내로 대체.
  내부 에러 내용은 노출하지 않는다.

## 5. 리버스 프록시(X-Forwarded-For) 신뢰 여부

- **위협**: Express의 `trust proxy`를 무조건 켜두면, 공격자가 `X-Forwarded-For`
  헤더를 조작해 매 요청마다 다른 IP인 것처럼 위장해 rate limit(로그인 시도 제한,
  전역 rate limit)을 우회할 수 있다. 반대로 실제로 ngrok 등 프록시 뒤에서 켜지
  않으면 모든 요청이 프록시의 IP 하나로 집계되어 정상 사용자끼리 rate limit을
  공유하게 된다.
- **선택**: 기본값은 `false`로 두고, `TRUST_PROXY=true` 환경변수를 명시적으로
  설정했을 때만 `app.set('trust proxy', 1)`이 적용되도록 했다 (`src/config/env.js`,
  `src/app.js`). 배포/터널링 환경에 따라 운영자가 의식적으로 선택하게 만든 것이다.

## 처음부터 반영한 기본 보안 조치 (참고)

아래 항목들은 "발견 후 수정"이 아니라 각 기능 구현 시점에 처음부터 적용한 것들이다.
전체 목록과 확인 방법은 `checklist.md` 참고.

- 비밀번호 bcrypt 해시, 평문 미저장
- 모든 DB 쿼리 파라미터 바인딩 (SQL Injection 방지)
- EJS 자동 이스케이프 + 채팅 클라이언트 textContent 사용 (XSS 방지)
- 모든 상태변경 요청 CSRF 토큰 검증
- 상품 수정/삭제, 관리자 라우트 등 서버측 소유자/권한 재검증 (IDOR 방지)
- 로그인 rate limit, 회원가입시 mass assignment(role/balance) 차단
- 파일 업로드 확장자/MIME/크기 제한, 랜덤 파일명
- 송금 로직의 원자적 트랜잭션 처리 및 잔액 음수 방지 DB 제약
- 관리자 초기 비밀번호 하드코딩 금지 (환경변수 필수, 미설정시 시드 실패)
