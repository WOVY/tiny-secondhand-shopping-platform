# 개발 과정에서 발견한 보안 약점과 수정 내역

이 프로젝트는 처음부터 시큐어 코딩을 염두에 두고 작성했지만(비밀번호 해시, 파라미터
바인딩, CSRF 토큰, 입력 검증 등을 기능 구현과 동시에 적용), 실제로 기능을 서로 연결하고
테스트하는 과정에서 아래와 같은 구체적인 약점들을 발견해 수정했다. "예시 코드 없이
처음부터 만들어도 보안 요소를 놓치기 쉽다"는 강의 취지에 맞춰, 실제로 겪은 문제 위주로
기록한다.

## 1. 세션 고정 방지 도입이 CSRF 보호를 깨뜨린 문제

- **증상**: 로그인/회원가입 성공시 세션 고정(session fixation) 공격을 막기 위해
  `req.session.regenerate()`로 세션 ID를 재발급하도록 구현했는데, 그 직후 첫 요청이
  전부 403(CSRF 오류)으로 실패했다.
- **원인**: CSRF 토큰(`csrf-csrf`)이 세션 ID에 해시로 묶여 있는데, 세션이 재발급되며
  기존 `csrf-token` 쿠키는 새 세션 ID 기준으로는 더 이상 유효하지 않게 되었다.
- **Before**: 세션 재발급 후 이전 CSRF 쿠키를 그대로 둠 → 다음 요청에서 검증 실패.
- **After**: CSRF 토큰 발급 미들웨어(`src/app.js`)에서 `generateToken()` 검증 실패를
  `try/catch`로 감지해, 실패시 새 세션 기준으로 토큰을 즉시 재발급하도록 수정
  (`src/middleware/csrf.js`). 로그인/로그아웃/회원가입 전 시나리오를 curl로 재검증했다.

## 2. helmet 기본 CSP가 인라인 이벤트 핸들러를 막아 삭제 확인이 조용히 깨짐

- **증상**: 상품/유저/신고 삭제 버튼에 `onsubmit="return confirm(...)"` 인라인 핸들러를
  사용했는데, helmet의 기본 CSP(`script-src 'self'`)가 인라인 스크립트(속성 포함)를
  차단해 확인 다이얼로그가 뜨지 않았다. CSRF나 인가 로직 자체는 안전했지만, 우리가 이미
  적용해 둔 보안 헤더와 UI 코드가 서로 충돌한 사례였다.
- **Before**: `onsubmit="return confirm('삭제하시겠습니까?');"` 인라인 속성.
- **After**: `data-confirm` 커스텀 속성 + 외부 스크립트(`public/js/confirm-forms.js`)의
  `addEventListener`로 교체. 동시에 인라인 `style="..."` 속성도 CSS 클래스로 옮기고,
  helmet CSP의 `style-src`에서 `unsafe-inline`을 명시적으로 제거해 CSP를 더 엄격하게
  만들었다.

## 3. multer 1.x의 알려진 취약점

- **증상**: 초기 의존성 설치시 `npm install`이 `multer@1.4.5-lts.2`에 대해
  "다수의 취약점이 있으며 2.x에서 패치됨"이라는 경고를 출력했다.
- **Before**: `multer: ^1.4.5-lts.1`
- **After**: `multer: ^2.0.1`로 즉시 상향하고 업로드 기능(확장자/크기 제한 포함)을
  재검증했다. `npm audit` 결과 0 vulnerabilities로 확인.

## 4. 관리자의 유저 삭제가 연관 데이터로 인해 서버 에러를 일으킬 뻔한 문제

- **증상**: 상품/거래내역/메시지/신고가 있는 유저를 관리자가 삭제하면, FK 제약
  (`PRAGMA foreign_keys = ON`)에 의해 SQLite가 예외를 던진다. 이를 그대로 두면
  관리자 페이지에서 500 에러가 나면서 내부 에러 메시지가 노출될 위험이 있었다.
- **Before**: 삭제 요청을 그대로 실행 → 예외 발생시 전역 에러 핸들러가 500 렌더링.
- **After**: `src/routes/admin.js`에서 삭제를 `try/catch`로 감싸, 실패시 "연관된
  데이터가 있어 삭제할 수 없습니다. 휴면 처리를 이용하세요"라는 안내로 대체.
  내부 에러 내용은 노출하지 않는다.

## 5. 리버스 프록시(X-Forwarded-For) 신뢰 여부

- **위협**: Express의 `trust proxy`를 무조건 켜두면, 공격자가 `X-Forwarded-For`
  헤더를 조작해 매 요청마다 다른 IP인 것처럼 위장해 rate limit(로그인 시도 제한,
  전역 rate limit)을 우회할 수 있다. 반대로 실제로 ngrok 등 프록시 뒤에서 켜지
  않으면 모든 요청이 프록시의 IP 하나로 집계되어 정상 사용자끼리 rate limit을
  공유하게 된다.
- **선택**: 기본값은 `false`로 두고, `TRUST_PROXY=true` 환경변수를 명시적으로
  설정했을 때만 `app.set('trust proxy', 1)`이 적용되도록 했다 (`src/config/env.js`,
  `src/app.js`). 배포/터널링 환경에 따라 운영자가 의식적으로 선택하게 만든 것이다.

## 6. 휴면 처리된 유저가 이미 연결된 소켓으로 계속 채팅할 수 있던 문제

- **발견 경위**: 최종 통합 검증(이슈 #11) 중, "신고 누적 → 자동 휴면 → 강제 로그아웃"이
  HTTP 요청(`requireAuth`)에는 잘 적용되지만, **이미 맺어진 Socket.IO 연결**에는 적용되지
  않는다는 것을 발견했다. 소켓 인증은 `connection` 이벤트 시점에 한 번만 검사했기 때문에,
  이미 채팅 중이던 사용자가 신고로 휴면 처리된 이후에도 같은 소켓으로 전체채팅/1:1채팅
  메시지를 계속 보낼 수 있었다. 실제로 재현 테스트(소켓 연결 → 신고 5회로 휴면 처리 →
  기존 소켓으로 메시지 전송)에서 메시지가 DB에 정상적으로 저장되는 것을 확인했다.
- **원인**: `src/sockets/chat.js`가 연결 시점에만 `user.status`를 확인하고, 이후
  메시지 이벤트 핸들러에서는 재확인하지 않았다.
- **Before**: `io.on('connection', ...)` 시점에만 상태 확인.
- **After**: 각 메시지/방참가 이벤트 핸들러(`chat:global:message`, `chat:dm:join`,
  `chat:dm:message`) 시작 부분에서 최신 유저 상태를 다시 조회하는 `isStillActive()`를
  추가. 휴면 상태로 확인되면 그 자리에서 소켓을 강제 종료한다. 동일한 재현 시나리오로
  재검증했고, 이번에는 소켓이 즉시 disconnect되고 메시지가 저장되지 않음을 확인했다
  (`testing.md` 참고).
- **교훈**: 여러 프로토콜(HTTP + WebSocket)이 같은 "인증 상태"를 공유할 때는, 상태 변경이
  모든 연결 방식에 실시간으로 반영되는지 각각 확인해야 한다. 기능 단위 테스트만으로는
  놓치기 쉽고, 기능을 결합한 통합 테스트에서야 드러난 문제였다.

## 7. 멀티파트 폼(이미지 업로드)에서 CSRF 검증이 항상 실패하던 문제

- **발견 경위**: 실제 사용자 테스트 중 로그인 후 상품 등록(`/products/new`)을 시도하면
  매번 403("요청이 거부되었습니다")이 발생한다는 보고를 받았다.
- **원인**: 상품 등록/수정 폼은 이미지 업로드 필드 때문에 `enctype="multipart/form-data"`를
  쓰는데, `express.urlencoded`/`express.json`은 이 형식을 파싱하지 않는다. 반면 CSRF 검증
  미들웨어(`doubleCsrfProtection`)는 전역으로 먼저 실행되며 `req.body._csrf`를 읽으려
  했는데, 이 시점에는 아직 아무것도 파싱되지 않아 값이 항상 비어 있었다. 실제로 바디를
  파싱하는 multer(`handleImageUpload`)는 라우트 내부에 있어 CSRF 검증보다 나중에 실행되고
  있었다 — 순서가 뒤바뀐 것이다.
- **Before**: 전역 CSRF 미들웨어 → (라우터 진입 후) multer 파싱. 파싱 전에 검증하므로
  멀티파트 요청은 정상적인 요청도 전부 거부됨.
- **After**: `src/app.js`에서 `/products/new`, `/products/:id/edit` 두 경로에 한해
  multer 파싱을 전역 CSRF 검증보다 먼저 실행하도록 이동. 로그인하지 않은 요청은 굳이
  파일을 디스크에 먼저 쓰지 않도록 세션 존재 여부를 먼저 확인한 뒤에만 파싱하도록
  했다. 라우트 내부에 남아있던 중복 multer 호출은 제거했다(그대로 두면 이미 소비된
  스트림을 다시 읽으려다 오류/행이 남).
- **검증**: 별도 포트/DB로 재현 — 수정 전에는 파일 유무와 무관하게 항상 403, 수정 후에는
  정상 등록(302, DB 반영) 및 실제 이미지 첨부 업로드까지 확인. CSRF 토큰을 아예 보내지
  않은 요청은 수정 후에도 여전히 403으로 거부됨을 재확인해, 우회가 아니라 순서만
  바로잡은 것임을 검증했다.
- **교훈**: 보안 미들웨어를 전역으로 걸 때는, 그 미들웨어가 의존하는 데이터(여기서는
  `req.body`)가 그 시점에 실제로 준비되어 있는지 요청 형식(Content-Type)별로 따져봐야
  한다. 기능 테스트를 라우트 단위로만 하면 놓치기 쉽고, 실사용자가 실제 폼으로
  테스트하다가 드러난 문제였다.

## 8. Bootstrap CDN이 기존 CSP에 막혀 스타일이 전혀 적용되지 않던 문제

- **발견 경위**: 이슈 #30(프론트엔드 UI 정리)에서 Bootstrap 5를 CDN(`cdn.jsdelivr.net`)
  방식으로 도입했는데, 실제로 페이지를 열어보니 CSS/JS가 전혀 적용되지 않은 상태로
  렌더링되었다.
- **원인**: `src/app.js`의 helmet CSP가 `style-src`/`script-src`를 `'self'`로 엄격하게
  제한하고 있어(2번 항목 참고), 외부 도메인인 CDN에서 불러오는 리소스는 브라우저가 그
  자리에서 차단했다. 인라인 스타일/스크립트뿐 아니라 신뢰하지 않은 외부 출처의 리소스도
  같은 정책으로 막힌 것으로, 이미 적용해 둔 보안 헤더와 새로 추가한 UI 코드가 다시 한
  번 충돌한 사례였다.
- **Before**: `views/partials/head.ejs`/`foot.ejs`에서
  `https://cdn.jsdelivr.net/npm/bootstrap@5.3.3/...` CDN 링크로 CSS/JS를 로드.
- **After**: CSP를 완화(외부 도메인 허용)하는 대신, Bootstrap 5.3.3 파일을
  `public/css/bootstrap.min.css`, `public/js/bootstrap.bundle.min.js`로 직접 받아
  자체 호스팅하도록 변경. 기존 `style-src`/`script-src 'self'` 정책을 그대로 유지하면서
  외부 CDN 의존성(공급망 공격 표면)도 새로 추가하지 않는 방향을 선택했다.
- **교훈**: 새 프론트엔드 라이브러리를 CDN으로 추가할 때는, 이미 적용된 CSP 정책과
  충돌하지 않는지 먼저 확인해야 한다. 정적 자산을 직접 서빙하는 것이 가능하다면, CSP를
  느슨하게 만드는 것보다 자체 호스팅이 더 안전한 기본 선택지다.

## 처음부터 반영한 기본 보안 조치 (참고)

아래 항목들은 "발견 후 수정"이 아니라 각 기능 구현 시점에 처음부터 적용한 것들이다.
전체 목록과 확인 방법은 `checklist.md` 참고.

- 비밀번호 bcrypt 해시, 평문 미저장
- 모든 DB 쿼리 파라미터 바인딩 (SQL Injection 방지)
- EJS 자동 이스케이프 + 채팅 클라이언트 textContent 사용 (XSS 방지)
- 모든 상태변경 요청 CSRF 토큰 검증
- 상품 수정/삭제, 관리자 라우트 등 서버측 소유자/권한 재검증 (IDOR 방지)
- 로그인 rate limit, 회원가입시 mass assignment(role/balance) 차단
- 파일 업로드 확장자/MIME/크기 제한, 랜덤 파일명
- 송금 로직의 원자적 트랜잭션 처리 및 잔액 음수 방지 DB 제약
- 관리자 초기 비밀번호 하드코딩 금지 (환경변수 필수, 미설정시 시드 실패)
