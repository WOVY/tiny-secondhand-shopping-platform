# 유지보수

실제로 기능을 사용해보며 발견한 불편한 점, 알려진 제한사항, 향후 개선 아이디어를 기록한다.

## 알려진 제한사항

- **세션 스토어가 메모리 기반(MemoryStore)**: 서버를 재시작하면 모든 로그인 세션이
  초기화된다. "tiny" 규모의 단일 프로세스 로컬 실행을 전제로 한 선택이며, 실제 운영
  환경으로 확장한다면 `connect-sqlite3` 등 영속 스토어로 교체가 필요하다.
- **자동화된 테스트는 smoke test 수준**: `npm run smoke`(`scripts/smoke-test.js`)로 회원가입
  →로그인→상품등록→조회→로그아웃 골든 패스와 CSRF/중복 아이디/관리자 인가 등 핵심 보안
  동작을 자동 검증할 수 있다. 다만 채팅(Socket.IO), 신고 누적에 따른 자동 차단/휴면, 송금,
  관리자 CRUD 등 `testing.md`에 기록된 나머지 시나리오는 여전히 수동 curl/socket.io-client
  검증에 의존한다. Jest + supertest 조합으로 커버리지를 넓히면 좋을 것이다.
- **관리자 페이지에 페이지네이션 없음**: 유저/상품/신고 목록을 전부 한 번에 불러온다.
  데이터가 많아지면 페이지네이션 또는 커서 기반 조회가 필요하다.
- **알림 기능 없음**: 신고 처리 결과, 송금 수신, 새 메시지 등에 대한 알림(이메일/푸시/
  뱃지 등)이 없다. 지금은 `/chat`에 접속해 대화 목록을 직접 확인해야 새 메시지를 알 수
  있다.
- **관리자의 전체 거래내역(송금) 조회 기능 없음**: "관리자가 전체 데이터를 관리한다"는
  요구사항을 넓게 해석하면 포함될 수 있었으나, 검토 결과 타인 간 송금 내역(금액/상대방)은
  당사자 외에는 노출하지 않는 것이 개인정보 보호 관점에서 더 적절하다고 판단해 의도적으로
  구현하지 않기로 확정했다. 각 유저는 본인의 거래내역(`/transfer`)만 조회할 수 있다.

## 향후 개선 아이디어

- 상품 이미지 다중 업로드 및 썸네일 생성
- 채팅 메시지 페이지네이션(무한 스크롤)
- 관리자 대시보드에 통계(가입자 수, 거래액, 신고 처리율 등) 추가
- 리프레시 토큰 없이도 세션 만료를 사용자에게 안내하는 UX 개선
- 배포 환경(HTTPS 종료 지점, 영속 세션 스토어, 로그 수집)을 고려한 프로덕션 설정 분리

## 유지보수 절차

1. 사용해보며 불편하거나 요구사항과 다른 부분을 이 문서에 기록
2. 원인이 요구사항 자체의 문제인지, 설계의 문제인지, 구현의 문제인지 확인
3. 해당하는 단계(`requirements.md` / `design.md` / 코드)로 돌아가 수정
4. 수정 후 `testing.md`의 관련 테스트 케이스를 다시 확인
