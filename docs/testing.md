# 테스트

체크리스트는 `checklist.md` 참고. 이 문서는 실제로 수행한 테스트 케이스와 결과를 정리한다.
모든 테스트는 로컬 서버(`npm start`)를 띄운 뒤 `curl`(세션 쿠키 유지) 및
`socket.io-client`(채팅)로 직접 실행했다.

## 골든 패스

기능별로 개별 검증한 뒤(이슈 #3~#8), 최종 검증(이슈 #11)에서 하나의 연속된 시나리오로
다시 실행했다: 회원가입(2명) → 상품 등록 → 검색으로 조회 → 상세페이지에서 채팅/신고 링크
확인 → 본인 상품 수정 → 실제 송금(구매 대금 지불 시나리오) → 상품 신고(임계치 미만이라
계속 노출됨을 확인) → 관리자가 신고 처리완료 표시 → 전체채팅/1:1채팅 송수신. 모두 통과.

| 시나리오 | 결과 |
|---|---|
| 회원가입 → 세션 로그인 상태 반영 | ✅ |

| 마이페이지 소개글 수정 | ✅ |

| 상품 등록 → 목록/상세 조회 | ✅ |
| 상품 검색(텍스트/가격 범위) | ✅ |
| 본인 상품 수정/삭제 | ✅ |
| 전체 채팅 송수신 | ✅ |
| 1:1 채팅 송수신, 새로고침 후 이력 복원 | ✅ |
| 상품/유저 신고 → 사유 필수 확인 | ✅ |
| 신고 누적(임계치 도달) → 자동 차단/휴면 | ✅ |

## 자동화된 smoke test

위 표의 시나리오는 대부분 일회성 수동 검증(curl/socket.io-client)이었지만, 그중 회귀가
발생하기 쉬운 핵심 경로는 `npm run smoke`(`scripts/smoke-test.js`)로 자동화해 뒀다. 별도
DB(`data/smoke-test.sqlite`)와 포트로 실제 서버를 띄운 뒤 fetch로 다음을 확인하고, 실패시
`exit code 1`을 반환한다(CI 연결 가능):

- 골든 패스: 회원가입 → 로그인 상태 확인 → 상품 등록 → 상세 페이지 조회 → 로그아웃 → 재로그인
- CSRF 토큰 없는 `POST /register` 거부(403)
- 중복 아이디 회원가입 거부(400)
- 일반 유저의 `GET /admin` 접근 거부(403)
- 잘못된 비밀번호 로그인 거부(400)

일부러 `admin.js`의 403 분기를 200으로 바꿔 스크립트가 실제로 실패(1개 FAIL, exit 1)하는
것을 확인한 뒤 원복해, 이 smoke test가 회귀를 실제로 잡아낸다는 것을 검증했다.
