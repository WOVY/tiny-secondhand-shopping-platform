# 요구사항 분석

## 배경 및 목표

시큐어 코딩 실습 프로젝트. "중고거래가 가능한 플랫폼(Tiny Second-hand Shopping Platform)"을 처음부터 구현하면서, 기능 요구사항을 만족시키는 동시에 개발 과정에서 보안 약점을 스스로 찾아 제거하는 것이 목표다.

요구사항은 두 그룹으로 나눈다.

## 그룹 A — 상세 설계가 주어진 요구사항

- 사람들이 플랫폼에 가입할 수 있어야 한다
  - 로그인/회원가입 페이지, 아이디 중복 불가, 유저 정보는 DB로 관리
  - 마이페이지(소개글/비밀번호 수정), 사용자 조회
- 상품을 올리고 볼 수 있어야 한다
  - 상품 등록 페이지, 본인 상품 관리, 목록은 이름만 노출·상세는 클릭시 조회
  - 상품 정보(상품명/가격/사진)는 DB로 관리, 등록된 상품은 누구나 조회 가능
- 플랫폼 사용자들끼리 소통이 가능해야 한다
  - 전체 유저 대상 채팅 + 1:1 채팅
- 악성 유저나 상품을 차단해야 한다
  - 신고 기능(사유 필수), 일정 횟수 이상 신고된 상품은 자동 차단, 유저는 자동 휴면

## 그룹 B — 직접 설계가 필요했던 요구사항

과제 명세에 상세 설계가 주어지지 않아 아래와 같이 자체적으로 요구사항을 도출했다.

- **유저간 송금**: 실제 결제망 연동 없는 내부 포인트/잔액 시스템으로 정의. 가입시 데모용 초기 잔액을 지급해 실제로 송금 기능을 사용해볼 수 있게 했다. 상세 설계 근거는 `design.md` 참고.
- **상품 검색**: 제목/설명 텍스트 검색 + 최소/최대 가격 필터로 정의.
- **관리자 페이지**: role 기반 권한 모델을 도입하고, 유저(휴면처리/해제/삭제)·상품(차단/해제/삭제)·신고(조회/처리완료 표시) 세 영역을 관리자가 전부 다룰 수 있도록 정의.

## 비기능 요구사항

- **보안**: 비밀번호 해시, SQL Injection/XSS/CSRF 방지, 인가(IDOR) 검증, 세션 보안, 파일 업로드 검증, rate limit 등 — 상세는 `checklist.md`, `security-fixes.md` 참고
- **환경**: 별도 인프라 없이 로컬(Node.js + SQLite)에서 바로 실행 가능해야 함
- **문서화**: 개발 전 과정(요구사항→설계→구현→테스트→유지보수)과 보안 약점 개선 내역을 문서로 남길 것
